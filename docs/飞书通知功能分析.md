# 飞书通知功能完整分析

## 一、功能概述

JeriBlog 集成了飞书机器人通知功能，支持将关键业务事件实时推送到飞书群聊，方便管理员及时响应。

### 支持的通知类型

1. **新评论通知** (`comment_new`) - 用户发表新评论时通知管理员
2. **友链申请通知** (`friend_apply`) - 收到新的友链申请时通知管理员
3. **反馈投诉通知** (`feedback_new`) - 收到新的反馈投诉时通知管理员
4. **RSS订阅日报** (`rss_feed_daily`) - 每天早上9点推送订阅源的未读文章

### 通知渠道

- **站内信**：所有通知类型都会创建站内信
- **邮件**：评论、友链、反馈会发送邮件（如果用户配置了邮箱）
- **飞书**：评论、友链、反馈、RSS日报会推送到飞书群聊

## 二、架构设计

### 2.1 核心模块

```
server/
├── internal/service/notification.go    # 通知业务逻辑层
├── pkg/notification/
│   ├── service.go                      # 通知服务统一入口
│   ├── feishu.go                       # 飞书通知实现
│   └── email.go                        # 邮件通知实现
└── pkg/feishu/
    ├── client.go                       # 飞书API客户端
    └── card.go                         # 飞书卡片构建器
```

### 2.2 调用链路

```
业务触发 → NotificationService → notifier.Service → FeishuNotifier → feishu.Client → 飞书API
```

**示例：新评论通知流程**

1. 用户发表评论 → `CommentService.Create()`
2. 调用 `NotificationService.NotifyCommentToAdmins()`
3. 创建站内信 → 异步发送邮件 → 异步发送飞书
4. `notifier.Service.SendFeishu()` → `FeishuNotifier.Send()`
5. 构建卡片 → `feishu.Client.SendMessage()` → 调用飞书Webhook

### 3.4 RSS订阅日报 (`rss_feed_daily`)

**触发时机**：每天早上9点（定时任务）

**通知对象**：所有管理员

**触发条件**：**只有存在未读文章时才发送**

**卡片内容**：
- 标题：📰 RSS订阅日报
- 颜色：紫色
- 信息：最多20篇未读文章列表（序号、标题、链接、来源友链）
- 交互：
  - 按钮1：全部已读（标记所有文章为已读，带回调）
  - 按钮2：查看详情（跳转到管理后台RSS订阅列表）

**定时任务配置**：
```go
// 每小时刷新RSS订阅源
"0 0 * * * *"  → RefreshAllRssFeeds()

// 每天凌晨5点清理已删除的RSS文章
"0 0 5 * * *"  → CleanDeletedRssArticles()

// 每天早上9点推送RSS日报
"0 0 9 * * *"  → SendRssFeedDailyNotification()
```

**推送逻辑**：
```go
// 1. 统计未读文章数量
unreadCount := rssService.GetUnreadArticlesCount()

// 2. 如果没有未读文章，直接返回，不发送通知
if unreadCount == 0 {
    return nil
}

// 3. 获取最多20篇未读文章
articles := rssService.GetUnreadArticles(20)

// 4. 发送飞书通知
notificationService.NotifyRssFeedDaily(unreadCount, articles)
```

**文章状态管理**：
- `is_read = false`：未读文章（新增、更新、删除都标记为未读）
- `is_deleted = true`：已删除的文章（RSS源中已不存在）

**代码位置**：
- 定时任务：`internal/task/rss.go`
- 业务逻辑：`internal/service/notification.go:152-167`
- 卡片构建：`pkg/feishu/card.go:234-248`

## 四、技术实现细节

### 4.1 飞书卡片 2.0 规范

使用飞书消息卡片 2.0 规范，支持丰富的交互元素：

**卡片结构**：
```json
{
  "schema": "2.0",
  "config": {
    "update_multi": true,
    "style": {
      "text_size": {
        "normal_v2": {
          "default": "normal",
          "pc": "normal",
          "mobile": "heading"
        }
      }
    }
  },
  "header": {
    "template": "blue",
    "title": {
      "content": "标题",
      "tag": "plain_text"
    }
  },
  "body": {
    "direction": "vertical",
    "elements": [
      {
        "tag": "markdown",
        "content": "内容"
      },
      {
        "tag": "button",
        "text": {
          "content": "按钮",
          "tag": "plain_text"
        },
        "type": "primary",
        "multi_url": {
          "url": "https://example.com"
        }
      }
    ]
  }
}
```

**支持的元素类型**：
- `markdown`：Markdown格式文本
- `input`：输入框（支持交互回调）
- `button`：按钮（支持URL跳转和交互回调）

**颜色模板**：
- `blue`：蓝色（评论通知）
- `green`：绿色（友链申请）
- `orange`：橙色（反馈投诉）
- `purple`：紫色（RSS日报）

### 4.2 异步发送机制

飞书通知采用异步发送，不阻塞主业务流程：

```go
// internal/service/notification.go:316-318
if s.notificationSvc != nil {
    go s.sendFeishu(notifType, title, content, link, data)
}
```

**优点**：
- 不影响用户体验（评论、申请等操作立即返回）
- 飞书API故障不影响核心业务
- 提高系统吞吐量

**注意事项**：
- 异步发送失败只记录日志，不抛出错误
- 无法保证消息一定送达（网络故障、飞书服务异常等）

### 4.3 错误处理

```go
// pkg/notification/feishu.go:33-36
func (f *FeishuNotifier) Send(data Data) error {
    if f.client == nil || !f.client.IsEnabled() {
        return nil  // 未配置飞书，静默跳过
    }
    // ...
}
```

```go

    logger.Warn("发送飞书通知失败: %v", err)  // 只记录警告，不中断流程
```

## 五、RSS订阅日报详细说明

### 5.1 工作流程

```
每小时刷新RSS → 检测文章变更 → 标记未读 → 每天9点统计 → 有未读则推送
```

### 5.2 文章变更检测

**新增文章**：
- RSS源中出现新的文章GUID
- 创建记录，`is_read = false`

**更新文章**：
- RSS源中已存在的GUID，但标题/内容/发布时间变化
- 更新记录，`is_read = false`（重新标记为未读）

**删除文章**：
- RSS源中不再包含的GUID
- 标记 `is_deleted = true`，`is_read = false`（删除也算未读事件）

### 5.3 推送策略

**推送时间**：每天早上9点

**推送条件**：
```go
unreadCount := rssService.GetUnreadArticlesCount()
if unreadCount == 0 {
    return nil  // 没有未读文章，不发送通知
}
```

**推送内容**：
- 最多显示20篇未读文章
- 按发布时间倒序排列（最新的在前）
- 每篇文章显示：序号、标题（带链接）、来源友链

**示例卡片**：
```
📰 RSS订阅日报

1. 《Go 1.23 新特性解析》 张三的博客
2. 《Docker 容器化最佳实践》 李四的技术站
3. 《Vue 3 组合式API详解》 王五的前端笔记

[全部已读] [查看详情]
```

### 5.4 已读管理

**手动标记已读**：
- 管理后台RSS订阅列表中点击"标记已读"
- 飞书卡片中点击"全部已读"按钮

**自动清理**：
- 每天凌晨5点清理已删除且已读的文章记录
- 保留未读的删除记录（等待管理员确认）

## 六、常见问题

### Q1: 为什么没有收到飞书通知？

**排查步骤**：
1. 检查 `.env` 中 `FEISHU_WEBHOOK_URL` 是否配置
2. 检查飞书机器人是否被移出群聊
3. 查看日志中是否有"发送飞书通知失败"错误
4. 确认触发条件是否满足（如RSS日报需要有未读文章）

### Q2: RSS日报为什么没有推送？

**可能原因**：
1. 没有未读文章（推送条件不满足）
2. 所有RSS订阅源都没有更新
3. 定时任务未正常运行
4. 飞书机器人配置错误

**验证方法**：
```bash
# 查看定时任务日志
grep "SendRssFeedDailyNotification" logs/app.log

# 查看未读文章数量
# 在管理后台 RSS订阅列表中查看
```

### Q3: 如何测试飞书通知？

**方法1：触发真实事件**
- 发表一条评论 → 触发评论通知
- 提交一个友链申请 → 触发友链通知
- 提交一个反馈投诉 → 触发反馈通知

**方法2：手动调用API**
```go
// 在代码中临时添加测试代码
notificationService.NotifyCommentToAdmins(ctx, userID, comment, "测试页面", nil)
```

### Q4: 飞书通知失败会影响业务吗？

**不会**。飞书通知采用异步发送，失败只记录日志，不影响：
- 评论发表成功
- 友链申请提交成功
- 反馈投诉提交成功
- 站内信和邮件通知正常发送

## 八、总结

JeriBlog 的飞书通知功能设计合理，具有以下特点：

**优点**：
- ✅ 架构清晰，职责分离
- ✅ 异步发送，不阻塞业务
- ✅ 优雅降级，配置灵活
- ✅ 卡片美观，交互友好
- ✅ RSS日报智能推送（只在有未读时发送）

**改进建议**：
- 考虑添加通知频率限制（防止刷屏）
- 考虑添加通知发送统计（成功率、失败率）
- 考虑支持更多飞书交互功能（如表单提交、审批流程）

**核心价值**：
- 提高管理员响应速度
- 减少后台登录频率
- 提升运营效率
